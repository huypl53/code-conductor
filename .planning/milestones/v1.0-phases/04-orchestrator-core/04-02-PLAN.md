<feature>
  <name>DependencyScheduler with graphlib</name>
  <files>packages/conductor-core/src/conductor/orchestrator/scheduler.py, packages/conductor-core/tests/test_scheduler.py</files>
  <behavior>
    CORD-04 behaviors:
    - DependencyScheduler(graph) where graph is dict[str, set[str]] (task_id -> dependency_ids)
    - get_ready() returns task IDs whose dependencies are all satisfied
    - Tasks with no dependencies appear in first get_ready() call
    - After done(task_id), dependents of that task become ready
    - is_active() returns True while tasks remain to schedule/complete
    - is_active() returns False after all tasks are done()
    - Circular dependency raises CycleError with cycle list
    - Empty graph (no tasks) works without error: is_active() returns False
    - Single task with no deps: get_ready() returns it, done() completes, is_active() False
    - Diamond dependency (A->C, B->C, C->D): A,B ready first, C after both done, D after C done
  </behavior>
  <implementation>
    DependencyScheduler wraps graphlib.TopologicalSorter exactly per research Pattern 2:
    - __init__ takes dict[str, set[str]], builds TopologicalSorter, calls prepare()
    - Catches GraphCycleError from prepare(), re-raises as CycleError with cycle path from exc.args[1]
    - get_ready() delegates to _sorter.get_ready()
    - done(task_id) delegates to _sorter.done(task_id)
    - is_active() delegates to _sorter.is_active()

    Import CycleError from conductor.orchestrator.errors. If errors.py doesn't exist yet (parallel wave), create a minimal stub with just CycleError.
  </implementation>
</feature>

<feature>
  <name>File ownership validation</name>
  <files>packages/conductor-core/src/conductor/orchestrator/ownership.py, packages/conductor-core/tests/test_file_ownership.py</files>
  <behavior>
    CORD-05 behaviors:
    - validate_file_ownership(tasks) where tasks is list of (task_id: str, target_file: str) tuples
    - Returns dict[str, set[str]] mapping task_id to owned file set
    - Two tasks with same target_file raises FileConflictError with both task IDs and the overlapping files
    - Tasks with unique target_files returns clean ownership map
    - Empty task list returns empty dict
    - Single task returns single-entry dict
    - Three tasks where two conflict: error reports the conflicting pair, not the clean one
  </behavior>
  <implementation>
    validate_file_ownership per research Pattern 3:
    - Build ownership dict: {task_id: {target_file}} for each task
    - Pairwise comparison for overlaps
    - Raise FileConflictError on first overlap found
    - Return ownership map if no conflicts

    Import FileConflictError from conductor.orchestrator.errors.
  </implementation>
</feature>
